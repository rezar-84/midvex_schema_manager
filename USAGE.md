# Usage Guide

This guide covers the normal operating workflow for Midvex Schema Manager in Odoo.

## Setup

1. Install the addon.
2. Open **Website → Configuration → Structured Data → Global Website Schema**.
3. Create one global settings record per website.
4. Fill organization, website URL, logo, contact, address, language, and social-link fields.
5. Click **Validate** and review the **Preview** tab.

Use **Schema Manager** access for users who create or edit schema configuration. Use **Schema User** for read-only access.

## Create a Static Page Schema

Use static Page Schema records when one schema belongs to one specific website page or URL.

1. Open **Website → Configuration → Structured Data → Create Page Schema**.
2. Select a schema template first, such as `Article`, `Product`, `FAQPage`, `BreadcrumbList`, or `Service`.
3. Choose the render target:
   - **Website Page** for a selected Odoo website page.
   - **Custom URL** for a specific URL path such as `/about`.
   - **Global** only for intentional site-wide schema.
4. Set the language code, for example `en`, `tr`, or `de`.
5. Fill the generated field rows.
6. Click **Validate**.

Custom URL records are unique by normalized URL, language, schema type, and website. You can create the same schema type on different URLs.

## Use One Schema Across Many Pages

Use **Model Mappings** when one schema structure should render across many records and values should be filled automatically.

1. Open **Website → Configuration → Structured Data → Model Mappings**.
2. Create a mapping for a supported model, such as `product.template`, `blog.post`, `website.page`, `event.event`, `slide.channel`, or `res.partner`.
3. Select a schema template.
4. Add mapping lines from schema dot-paths to Odoo fields or safe tokens.
5. Save and test a matching frontend page.

Examples:

| Schema field | Source |
|---|---|
| `name` | Odoo field `name` |
| `description` | Odoo field `description_sale` |
| `url` | Token `{{ current.canonical_url }}` |
| `image` | Token `{{ product.image_url }}` |
| `author.name` | Token `{{ blog.author }}` |

No Python expressions are evaluated in tokens.

## Import / Export Configuration

Use **Import / Export Configs** to move reusable configuration between Odoo databases.

1. Open **Website → Configuration → Structured Data → Import / Export Configs**.
2. Choose **Export** or **Import**.
3. Choose the format:
   - **Excel Workbook (`.xlsx`)**
   - **CSV Files (`.zip`)**
4. Select which configuration to include:
   - Global Settings
   - Templates
   - Model Mappings
5. Export and download the generated file, or upload a file generated by the wizard and import it.

The importer updates existing records by stable keys:

| Data | Matching key |
|---|---|
| Global Settings | Resolved website |
| Templates | `name + schema_type` |
| Model Mappings | Mapping name |
| Mapping Lines | Replaced for the imported mapping |

Page Schema records are not exported by this wizard because they are page/content instances. For many similar pages, prefer Model Mappings. For a small number of exact static records, use Odoo's standard import on **Page Schemas**.

## Validate and Preview

Use **Validate** on each schema record before publishing. The validator checks:

- JSON-LD structure
- Required template fields
- URL format
- Language code format
- FAQ and breadcrumb completeness
- Duplicate schema targets

Use these buttons for external checks:

- **Open Schema Markup Validator**
- **Open Google Rich Results Test**

## Language Workflow

Create one schema record per language. Use **Create for All Languages** from a source schema record to duplicate it for active website languages.

Records are rendered only when their language matches the current frontend request language. Leave the language blank only when the schema should apply to every language.

## Batch Operations

Open **Website → Configuration → Structured Data → Batch Operations** for maintenance tasks:

- Validate all active schemas
- Regenerate backend previews
- Create translated schema records for active website languages

These operations update backend metadata and cached previews. Public rendering still builds schema safely at request time.

## Troubleshooting

If import reports a duplicate Page Schema, check the target type and URL:

- **Custom URL** records must have a target URL.
- URLs are normalized, so `/page`, `/page/`, and `https://domain.com/page/` are the same target.
- The same schema type is allowed on different URLs.

If a schema should apply automatically to many product, blog, event, or page records, create a Model Mapping instead of duplicating static Page Schema records.
